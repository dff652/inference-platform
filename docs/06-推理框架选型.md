# 推理框架选型

## 最后更新：2026-03-16

## 重要说明

本文档主要记录 GPU 推理框架的选型分析和中长期目标架构。

截至 2026-03-16，仓库当前真实落地状态为：

- GPU 服务已实现为 `transformers` 常驻 FastAPI 服务
- 平台侧通过 OpenAI 兼容 HTTP 接口调用这些服务
- 真正切换到 vLLM 仍未完成，属于后续演进方向

当前实际状态请以 `README.md`、`docs/04-联调记录与环境配置.md`、`docs/05-开发进度与待办.md` 为准。

## 一、背景

当前平台通过 Celery + subprocess 调用 transformers 直接加载模型推理，存在以下问题：

- 每次任务启动都需加载模型（ChatTS-8B ~31s，Qwen-VL ~31s），无法常驻
- 不支持 continuous batching，并发请求只能排队
- 无 PagedAttention，显存利用率低
- 多机多卡扩展需要自建调度逻辑

需要选择推理框架，实现：模型常驻、高吞吐推理、多 GPU/多机部署、对外提供推理服务。

## 二、当前模型清单与特征

### 2.1 模型架构对比

| 模型 | 基座 | 架构类 | 输入类型 | 自定义代码 | 显存 |
|------|------|--------|----------|-----------|------|
| ChatTS-8B | Qwen2.5 微调 | Qwen3TSForCausalLM | 文本 + 时序数组 | 3 个文件 + monkey patch | ~7.6GB (4-bit) |
| Qwen3-VL-8B | Qwen3-VL | Qwen3VLForConditionalGeneration | 文本 + 图片 | trust_remote_code | ~18GB (bf16) |
| Qwen3.5-27B | Qwen3.5 | Qwen3_5ForConditionalGeneration | 文本 + 图片 | trust_remote_code | ~15GB (4-bit) |
| Timer/SunDial | 清华 Timer | AutoModelForCausalLM | 纯时序数组 | trust_remote_code | ~4-8GB |

### 2.2 ChatTS 自定义组件说明

ChatTS 基于 Qwen2.5 微调，额外增加了时序理解能力，核心自定义：

| 文件 | 功能 |
|------|------|
| `modeling_qwen3_ts.py` | `TimeSeriesEmbedding`（MLP patch 编码）+ `_merge_input_ids_with_time_series_features`（时序嵌入注入 token 序列） |
| `processing_qwen3_ts.py` | `Qwen3TSProcessor`（时序归一化 + `<ts><ts/>` 占位符替换） |
| `configuration_qwen3_ts.py` | 时序相关配置（patch_size=8, embedding_dim=16 等） |
| `patch_transformers.py` | 运行时 monkey patch 修复 `_update_causal_mask` 维度不匹配 |

### 2.3 Qwen-VL 推理流程

时序数据 → matplotlib 渲染为图片 → Qwen-VL 视觉理解 → 输出异常段 JSON。标准 VL 模型架构，无深度自定义。

### 2.4 Timer 推理流程

时序数据 → 滚动窗口预测 → 残差计算 → 阈值检测异常。非标准 LLM 生成任务，基于预测残差做统计检测。

## 三、主流推理框架评估

### 3.1 框架概览

| 框架 | 核心优势 | 多卡 | 多机 | 模型服务 API | LoRA 热加载 |
|------|---------|------|------|-------------|------------|
| **vLLM** | PagedAttention + continuous batching | tensor parallel | Ray 集群 | OpenAI 兼容 | 支持 multi-LoRA |
| **SGLang** | RadixAttention + 前缀共享 | tensor parallel | 多实例 | OpenAI 兼容 | 支持 |
| **TGI** | HuggingFace 生态 | tensor parallel | K8s 部署 | REST API | 有限 |
| **Triton** | 多框架通用（ONNX/TRT/PyTorch） | 多模型实例 | K8s 部署 | gRPC + REST | 不支持 |
| **Ray Serve** | Python 原生、灵活编排 | 自动调度 | 原生集群 | 自定义 | 不适用 |

### 3.2 各模型适配情况

| 框架 | ChatTS-8B | Qwen-VL (8B/27B) | Timer | 综合评价 |
|------|-----------|-------------------|-------|---------|
| **vLLM** | ✅ 官方插件支持 | ✅ 原生支持 | ❌ 非生成任务 | GPU 模型首选 |
| **SGLang** | ⚠️ 需自行适配 | ✅ 原生支持 | ❌ | 备选 |
| **TGI** | ❌ | ⚠️ VL 支持有限 | ❌ | 不推荐 |
| **Triton** | ⚠️ 需写 backend | ⚠️ 需写 backend | ⚠️ 需写 backend | 过重 |
| **Ray Serve** | ✅ 封装即可 | ✅ 封装即可 | ✅ 封装即可 | 灵活但无推理加速 |

### 3.3 关键发现：ChatTS 官方已支持 vLLM

ChatTS 官方仓库（[NetManAIOps/ChatTS](https://github.com/NetManAIOps/ChatTS)）提供了 vLLM 注册补丁：

```python
# 注册 ChatTS 自定义架构到 vLLM
import chatts.vllm.chatts_vllm

from vllm import LLM
llm = LLM(
    model="path/to/ChatTS-8B",
    tensor_parallel_size=2,
    max_model_len=6000,
    gpu_memory_utilization=0.95,
    limit_mm_per_prompt={"time_series": 10},
)
```

- 时序 embedding 通过 vLLM 多模态输入管道传入
- 支持 KV cache + AutoProcessor
- 支持 tensor parallel 多卡部署

## 四、vLLM vs SGLang 深度对比

在框架评估中，vLLM 和 SGLang 是最有竞争力的两个候选，均来自 UC Berkeley，架构理念不同但目标一致。

### 4.1 核心机制差异

| 维度 | vLLM | SGLang |
|------|------|--------|
| **核心技术** | PagedAttention（显存分页管理） | RadixAttention（前缀 KV cache 复用） |
| **调度方式** | continuous batching | continuous batching + 前缀感知调度 |
| **吞吐量** (H100) | ~12,500 tok/s | ~16,200 tok/s（**快 29%**） |
| **多轮对话** | 标准 | 快 10-20%（前缀共享优势） |
| **显存效率** | PagedAttention 减少碎片 | RadixAttention 复用已有 cache |
| **结构化输出** | guided decoding（outlines） | 压缩有限状态机（**更快的 JSON 约束生成**） |
| **LoRA** | multi-LoRA per request + VL tower 实验性支持 | multi-LoRA batching |
| **多卡** | tensor parallel + pipeline parallel | tensor parallel + data parallel + expert parallel |
| **多机** | Ray 集群 | 原生分布式 |

### 4.2 项目适配对比

| 维度 | vLLM | SGLang |
|------|------|--------|
| **ChatTS-8B** | ✅ 官方已提供 `chatts_vllm` 插件 | ❌ 无现成支持，需自行实现 model + processor + padding |
| **Qwen-VL** | ✅ 原生支持 | ✅ 原生支持 |
| **Qwen-VL LoRA** | ✅ 语言层全支持 + 视觉层实验性 | ✅ multi-LoRA batching |
| **自定义模型注册** | 插件 API，不改源码 | 需修改 SGLang 源码（model_config.py + conversation.py + processor） |
| **JSON 输出** | guided decoding | 压缩 FSM（更适合 ChatTS/Qwen 的 JSON 输出场景） |

### 4.3 RadixAttention 对本项目的价值

ChatTS 的推理特点是**同一个 prompt 模板 + 不同的时序数据**：

```
"我有一个长度为 768 的时间序列：<ts><ts/>。请识别该时间序列中所有异常或异常片段。"
```

每次请求 prompt 前缀完全相同，只有 `<ts>` 内的时序数据不同。RadixAttention 可缓存并复用公共前缀的 KV cache，批量推理场景下优势明显。

### 4.4 对比结论

| | vLLM | SGLang |
|---|---|---|
| **性能** | 良好 | **更优**（吞吐高 29%，JSON 输出更快） |
| **ChatTS 支持** | **开箱即用** | 需自行适配（工作量大） |
| **Qwen-VL 支持** | 成熟 | 成熟 |
| **生态/社区** | 最大，文档最全 | 快速增长，略小 |
| **长期潜力** | 持续迭代 | 架构更新，性能天花板更高 |

**决策：当前选 vLLM，关注 SGLang 演进。**

- ChatTS 官方只提供了 vLLM 插件，这是决定性因素
- vLLM 生态更成熟，遇到问题排查资料更多
- 当前规模（单机 2×22GB GPU）性能差异不明显，瓶颈在显存容量而非推理引擎
- 两者都暴露 OpenAI 兼容 API，未来切换 SGLang 时平台适配层不用改

## 五、选型结论

### 5.1 推荐方案：vLLM 统一 GPU 推理

| 模型 | 推理框架 | 部署方式 |
|------|---------|---------|
| ChatTS-8B | vLLM（chatts_vllm 插件注册） | vLLM server → OpenAI 兼容 API |
| Qwen-VL-8B/27B | vLLM（原生支持 + LoRA） | vLLM server → OpenAI 兼容 API |
| Timer/SunDial | transformers 直接调用 | 保持 subprocess 或封装为独立服务 |
| CPU 算法 | 无需框架 | subprocess 调旧项目脚本（保持现状） |

### 5.2 选择 vLLM 的理由

1. **覆盖面最广**：ChatTS（官方插件） + Qwen-VL（原生）均支持，不需要多框架混合
2. **性能优势**：PagedAttention 显存利用率提升 2-4x，continuous batching 吞吐提升 5-10x
3. **多卡多机原生支持**：tensor parallel（单机多卡）+ Ray 集群（多机）
4. **LoRA 热加载**：Qwen-VL 微调后 adapter 可在线切换，无需重启服务
5. **标准 API**：OpenAI 兼容接口，其他系统可直接对接
6. **社区活跃**：2026 年最主流的 LLM 推理框架，持续迭代

### 5.3 Timer 不用 vLLM 的原因

Timer 的推理方式是滚动窗口预测 + 残差阈值检测，不是标准的 LLM 文本生成（`model.generate()`），而是循环调用 forward pass 计算预测值与真实值的残差。vLLM 的核心优化（KV cache、continuous batching）对此类推理模式无效。

## 五、目标架构

### 5.1 部署拓扑

```
                    ┌─────────────────────────────────────────────┐
                    │              推理平台 API (:8100)             │
                    │     FastAPI + 任务调度 + 结果管理              │
                    └───────┬──────────┬──────────┬───────────────┘
                            │          │          │
                   ┌────────▼───┐ ┌────▼─────┐ ┌─▼──────────────┐
                   │ vLLM Server │ │ vLLM     │ │ Subprocess     │
                   │ ChatTS-8B  │ │ Qwen-VL  │ │ Timer + CPU 算法│
                   │ :8001      │ │ :8002    │ │                │
                   │ OpenAI API │ │ OpenAI   │ │ transformers   │
                   └────────────┘ │ API      │ └────────────────┘
                                  │ +LoRA    │
                                  └──────────┘
```

### 5.2 多机多卡扩展

| 场景 | 方案 | 配置 |
|------|------|------|
| 单机多卡 | tensor parallel | `tensor_parallel_size=N` |
| 多机多卡 | Ray 集群 + tensor parallel | `ray start --head` + 各节点 `ray start --address=head:6379` |
| 高吞吐扩容 | 多 vLLM 实例 + 负载均衡 | Nginx/HAProxy 反向代理多个 vLLM server |
| 多模型共存 | 独立 vLLM 实例 | 各模型独立端口、独立 GPU 分配 |

### 5.3 对外服务能力

vLLM server 原生提供 OpenAI 兼容 API，其他系统可直接调用：

```bash
# 启动 ChatTS 服务
VLLM_WORKER_MULTIPROC_METHOD=spawn \
vllm serve /path/to/ChatTS-8B \
  --tensor-parallel-size 2 \
  --port 8001 \
  --gpu-memory-utilization 0.95

# 启动 Qwen-VL 服务（含 LoRA adapter）
vllm serve Qwen/Qwen3-VL-8B-Instruct \
  --lora-modules anomaly-adapter=/path/to/lora_weights \
  --port 8002
```

调用方式：

```python
# 任何系统均可通过 OpenAI SDK 或 HTTP 调用
import openai
client = openai.OpenAI(base_url="http://server:8001/v1", api_key="unused")
response = client.chat.completions.create(
    model="ChatTS-8B",
    messages=[{"role": "user", "content": "分析时序异常..."}],
)
```

## 六、平台适配改造计划

### 6.1 执行适配器重构

当前 `CLIExecutorAdapter`（subprocess 调 run.py）→ 新增 `VLLMAdapter`（HTTP 调 vLLM API）：

```python
class ExecutorBackend(ABC):
    """推理后端抽象接口"""
    @abstractmethod
    async def execute(self, request: ExecutionRequest) -> ExecutionResult: ...

class VLLMBackend(ExecutorBackend):
    """GPU 模型：调用 vLLM OpenAI 兼容 API"""
    async def execute(self, request):
        # 1. 构建 prompt（ChatTS: 文本+时序 / Qwen: 文本+图片）
        # 2. HTTP POST → vLLM /v1/chat/completions
        # 3. 解析返回 JSON → ExecutionResult
        ...

class SubprocessBackend(ExecutorBackend):
    """Timer + CPU 算法：保持现有 subprocess 方式"""
    async def execute(self, request):
        # 现有 CLIExecutorAdapter 逻辑
        ...
```

### 6.2 路由逻辑

```python
# 模型 → 后端映射
BACKEND_MAPPING = {
    "chatts":          VLLMBackend(endpoint="http://localhost:8001"),
    "qwen":            VLLMBackend(endpoint="http://localhost:8002"),
    "timer":           SubprocessBackend(),
    "adtk_hbos":       SubprocessBackend(),
    "ensemble":        SubprocessBackend(),
    "wavelet":         SubprocessBackend(),
    "isolation_forest": SubprocessBackend(),
}
```

用户创建任务时只选算法，平台根据映射自动路由到对应后端，用户无需感知底层框架。

### 6.3 改造优先级

| 阶段 | 内容 | 依赖 |
|------|------|------|
| Phase 1 | Qwen-VL 切 vLLM serving | vLLM 安装 + Qwen-VL 启动验证 |
| Phase 2 | ChatTS 切 vLLM serving | chatts_vllm 插件验证 + 时序输入管道测试 |
| Phase 3 | 适配器抽象层重构 | VLLMBackend + SubprocessBackend 接口统一 |
| Phase 4 | 多机部署 | Ray 集群搭建 + tensor parallel 跨机验证 |

## 七、已知风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ChatTS vLLM 插件为实验性 | 可能存在兼容性问题 | 先在测试环境验证，保留 subprocess 回退路径 |
| Qwen3.5-27B LoRA 加载 bug | [vLLM #36372](https://github.com/vllm-project/vllm/issues/36372) | 优先使用 Qwen3-VL-8B，等待 bug 修复 |
| vLLM 多卡需要 NCCL | 多机通信依赖网络带宽 | 单机 tensor parallel 优先，多机按需启用 |
| Timer 无法用 vLLM | 需保持 subprocess 双轨运行 | 封装为独立 FastAPI 微服务作为备选 |
| vLLM 版本快速迭代 | 接口可能变动 | 锁定 vLLM 版本，定期评估升级 |
