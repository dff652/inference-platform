## 二、GPU 推理架构现状

### 2.1 当前 vLLM 使用边界

| 约束 | 详情 | 类型 |
|------|------|------|
| ChatTS 模型架构不支持 | `Qwen3TSForCausalLM`，vLLM 插件只有 Qwen2TS | 版本/模型 |
| Qwen3-VL 默认配置不可用 | `vllm 0.11.0` 可启动，但默认参数在 RTX 2080 Ti 上显存不足 | 显存/配置 |
| RTX 2080 Ti SM 7.5 | 不支持 flash-attn / bfloat16，需 `eager + float16` | 硬件 |
| Hugging Face 版本漂移 | `transformers 5.x` 会导致 tokenizer 不兼容，需锁回 4.57.6 | 依赖 |

**结论：**
- ChatTS 当前仍不适合切到真实 vLLM（`Qwen3TSForCausalLM` 无 vLLM 插件），保留 transformers 直接推理。
- Qwen3-VL 已正式切换到 `vLLM 0.11.0` 生产运行（systemd 服务常驻），受限配置下单机低并发场景稳定。
- 平台 HTTP 适配层统一兼容 transformers serve 和 `vllm serve`，无需区分后端实现。

  │   └── GPU 方法 → dispatcher.run_vllm() → HTTP 调 GPU 服务

## 四、已知技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| SQLite 并发写入限制 | Celery 多 worker 并发写 DB 可能冲突 | 一期单 worker；二期切 PostgreSQL |
| Qwen3.5-27B 显存不足 | 2×22GB 不够加载 | 代码已支持多卡和 4-bit，换 A100/4090 后可用 |
| Qwen3-VL vLLM 吞吐受限 | 当前需单并发 + 短上下文 + eager 模式 | 先用于验证/轻量场景，后续评估更大显存设备或继续调参 |
| ChatTS 暂不支持真实 vLLM | 无法统一到同一套 vLLM 运行时 | 暂保留 transformers 服务，等待上游支持或单独做插件验证 |
| GPU 服务无自动重启 | 进程意外退出后需手动重启 | ✅ 已解决：systemd Restart=always + platform.sh 管理 |

替代 `metadata.create_all()` 的自动建表方式，所有 model 字段变更通过迁移脚本管理。

```bash
# 修改 model 后生成迁移
cd backend
alembic revision --autogenerate -m "add xxx field to inference_task"

# 应用迁移
alembic upgrade head

# 查看当前版本
alembic current
```

**规则**：改 `models/*.py` 必须生成对应迁移文件一起提交，其他人 pull 后执行 `alembic upgrade head`。

#### 前端文件按领域拆分

```
src/api/                          src/utils/
├── client.js    ← 公共 axios     ├── task-constants.js  ← 任务相关常量
├── tasks.js     ← 推理任务 API    └── constants.js       ← re-export hub
├── models.js    ← 模型 + GPU API
├── configs.js   ← 配置 API
├── uploads.js   ← 上传 API
└── index.js     ← re-export hub（兼容旧引用）
```

各 Vue 文件已改为直接引用具体模块文件（如 `import { taskApi } from '../api/tasks'`），
不同人开发不同页面时不再修改同一个 API 文件。

### 6.3 Git 协作流程

```
main（保护分支，不直接 push）
├── feat/task-perf-tab         ← 人员A 开发 Performance Tab
├── feat/data-quality-precheck ← 人员A 开发数据质量预检
├── feat/evaluation-page       ← 人员B 开发模型评估页
├── feat/data-sources          ← 人员B 开发数据池页
└── fix/xxx                    ← bug 修复

合并规则：
1. models/ 和 schemas/ 的改动 → 必须互相 review
2. alembic/versions/ → 线性合并（先到先得，后来的 rebase）
3. 自己独占区的 .vue / .py 文件 → 自行合并
4. 合并前执行 npm run build + alembic upgrade head 验证
```

### 6.4 推理任务模块可立即分配的任务

以下任务完全在独占区内，不涉及 model/schema 共享区变更，可立即并行开发：

| 任务 | 涉及文件 | 说明 |
|------|----------|------|
| TaskDetail Performance Tab | `TaskDetail.vue` + `api/tasks.js` | 数据来自 `perf_logger.py` 的 JSON 文件，不改 DB |
| TaskCreate 数据质量预检 | `TaskCreate.vue` + 新建后端 API | 调用已有 `preprocessing.py` 函数，前端展示检测结果 |
| TimeSeriesChart 交互增强 | `TimeSeriesChart.vue` | 纯前端 D3 改动（双 Y 轴/多系列/键盘导航等） |
| CPU 算法端到端验证 | `algorithms/*` | 验证 Ensemble/Wavelet/IForest 全链路 |

**需要协商的任务**（涉及共享区）：

| 任务 | 涉及共享文件 | 协商点 |
|------|-------------|--------|
| 给 InferenceTask 加 perf 字段 | `models/inference_task.py` + `schemas/` | 先定字段名/类型，走 Alembic 迁移 |
| 新增评估结果表 | `models/` 新增文件 + `alembic/versions/` | 表结构设计先对齐 |

### 6.5 各模块并行开发可行性分析

| 模块 | 可否立即并行 | 前置条件 | 说明 |
|------|-------------|----------|------|
| 推理任务增强 | ✅ 可以 | 无 | 独占文件，不碰共享区 |
| 数据池页面 | ✅ 可以 | 无 | 后端 API 已就绪，新建 `DataSources.vue` + `api/data-sources.js` |
| 模型评估页面 | ⚠️ 需先定 schema | 需新建评估结果表 | 后端 `eval_metrics.py` 已有，需封装 API + 新 ORM 模型 |
| 数据标注页面 | ✅ 可以 | 无 | 纯前端 D3 标注器，独立 `Annotator.vue` |
| 模型中心增强 | ✅ 可以 | 无 | 独占 `ModelCenter.vue` + `api/models.js` |
| 用户管理 | ⚠️ 需先定认证方案 | JWT/API Key 设计 | 涉及新 model + 中间件，影响全局 |
| 模型微调 | ❌ 暂不建议 | 需 LLaMA-Factory 集成方案 | 依赖外部工具链，需先技术验证 |
| 系统设置 | ✅ 可以 | 无 | 独立 CRUD 页面，不影响其他模块 |

**结论**：当前版本下，推理任务 / 数据池 / 数据标注 / 模型中心 / 系统设置 共 5 个模块可立即并行开发，无需等待。模型评估和用户管理需先完成 schema 设计后再开工。
