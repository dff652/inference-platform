## 四、二期技术演进路径

### 关键前置决策：是否有 K8s 基础设施

- 有 K8s → 走 K8s + Volcano 路线
- 无 K8s → 走 Celery 多队列 + vLLM 路线（推荐）

### 方案对比

| 方案 | 多GPU | 多机 | 复杂度 | 适合场景 |
|------|-------|------|--------|----------|
| A: Celery + GPU 队列 | 按队列绑 GPU | 多机 worker | 低 | 任务独占 GPU，不需要模型常驻 |
| B: Ray Serve | 原生支持 | 原生集群 | 中 | 模型需要常驻内存，弹性伸缩 |
| C: vLLM 独立服务 | tensor parallel | 需配合调度层 | 中 | 大语言模型高吞吐推理 |
| D: K8s + Volcano | device plugin | 原生多机 | 高 | 已有 K8s 基础设施 |

### 推荐路径（无 K8s 场景）

```
二期 Phase 1: Celery 多队列 + GPU 亲和
  - 每台机器一个 worker，通过 CUDA_VISIBLE_DEVICES 绑定 GPU
  - 按 resource_profile 路由到对应队列
  - 改动最小，一期代码几乎不用动

二期 Phase 2: ChatTS/Qwen 切到 vLLM serving
  - 大模型常驻 GPU 内存，避免反复加载
  - executor_adapter 新增 vllm 类型，调 HTTP API
  - 吞吐量提升 5-10x

二期 Phase 3: 按需引入 Ray 或 K8s
  - 只在真正需要弹性伸缩时再上
```

### 二期启动条件

| 触发条件 | 指标 |
|----------|------|
| 并发瓶颈 | 单机 GPU 利用率持续 >80%，任务排队 >30min |
| 业务需求 | 需要多大模型并行，或单任务需多卡 tensor parallel |
| 平台稳定 | 一期任务成功率 >95%，无重大 bug |

建议时间线：一期上线 + 2~4 周稳定期 → 开始二期设计评审

## 五、一期为二期预留的扩展点

- `executor_type` 字段：当前 "cli"，二期加 "celery_worker" / "vllm" / "ray"
- `resource_profile` 字段：当前未用，二期映射到 GPU 队列
- `ExecutorAdapter` 独立类：二期新增 VLLMExecutorAdapter、RayExecutorAdapter
- `worker_node` 字段：二期记录任务分配到哪台机器

## 六、一期技术债务清单（二期必须解决）

| 债务 | 二期解决方案 |
|------|-------------|
| SQLite 不支持并发写入 | 切 PostgreSQL |
| 模型每次推理都加载 | vLLM serving 常驻内存 |
| Celery 调度能力有限 | GPU 亲和队列 或 自建调度器 |
| run.py 紧耦合 | 逐步拆解为独立服务 |
