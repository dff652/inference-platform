# 部署操作手册

## 最后更新

2026-03-16

## 1. 当前支持的部署方式

### 方式 A：单机手工部署

适合开发、联调、问题定位。

包含进程：

- Redis
- ChatTS GPU 服务
- Qwen GPU 服务
- FastAPI
- Celery worker
- 前端 dev server

### 方式 B：混合部署

适合当前仓库的准生产用法。

包含：

- Docker Compose 启动 `backend`、`celery_worker`、`redis`
- 宿主机单独启动 GPU 服务

限制：

- 不是全容器化
- worker 依赖宿主机挂载目录
- GPU 服务网络连通性需要手动保证

## 2. 单机手工部署

### 2.1 启动顺序

```bash
cd /home/dff652/dff_project/inference-platform
cp .env.example .env
```

1. 启动 Redis

```bash
redis-server --daemonize yes
```

2. 启动 GPU 服务

```bash
./services/start_vllm.sh all
```

3. 启动后端 API

```bash
cd backend
/home/dff652/miniconda3/envs/inference-platform/bin/uvicorn app.main:app --host 0.0.0.0 --port 8100
```

4. 启动 Celery worker

```bash
cd backend
/home/dff652/miniconda3/envs/inference-platform/bin/celery -A app.core.celery_app worker --loglevel=info
```

5. 启动前端

```bash
cd frontend
npm run dev
```

## 3. 混合部署

### 3.1 前提

- 已准备 `.env`
- 宿主机存在旧项目目录
- 宿主机存在 conda 安装目录
- 宿主机存在模型目录

### 3.2 启动

```bash
cd /home/dff652/dff_project/inference-platform
docker compose up -d
```

### 3.3 宿主机额外启动 GPU 服务

```bash
cd /home/dff652/dff_project/inference-platform
./services/start_vllm.sh all
```

### 3.4 关键说明

- [docker-compose.yml](/home/dff652/dff_project/inference-platform/docker-compose.yml) 不包含 GPU 服务容器
- `celery_worker` 会挂载宿主机 `OLD_PROJECT_PATH`、`MODEL_DIR`、`CONDA_PREFIX`
- 若容器内访问不到宿主机 `localhost:8001/8002`，应把 `VLLM_CHATTS_ENDPOINT`、`VLLM_QWEN_ENDPOINT` 改为宿主机可达地址

## 4. 开发模式 Compose

`docker-compose.dev.yml` 主要用于后端热更新开发，不是完整生产部署方案。

用途：

- 后端调试
- Celery 调试
- Redis 联调

限制：

- 文档和配置仍偏旧
- 没有描述当前 GPU 服务链路

## 5. 健康检查

部署完成后至少执行以下检查：

```bash
curl http://localhost:8100/health
curl http://localhost:8001/health
curl http://localhost:8002/health
curl http://localhost:8100/api/v1/models/vllm/status
curl http://localhost:8100/api/v1/inference/configs/algorithms
```

期望：

- `8100/health` 返回 `status=ok`
- `8001/health`、`8002/health` 返回模型已加载或正在加载
- `models/vllm/status` 能看到 `healthy=true/false`

## 6. 首次联调建议

顺序建议：

1. 先测 `health`
2. 再跑 `scripts/seed_data.py`
3. 再在前端创建一个 CPU 任务
4. 最后再测 GPU 任务

## 7. 常见故障

### 7.1 `Cannot connect to vLLM server`

原因：

- GPU 服务未启动
- `VLLM_*_ENDPOINT` 配置错
- 容器内无法访问宿主机地址

排查：

- 先 `curl` GPU 服务 `/health`
- 再检查 `.env`
- 再检查容器网络

### 7.2 Celery worker 执行失败

原因：

- `OLD_PROJECT_PATH` 不存在
- `OLD_PYTHON_PATH` / `GPU_PYTHON_PATH` 路径错误
- 宿主机挂载目录为空

排查：

- 检查 `.env`
- 检查 Compose 卷挂载
- 检查 worker 日志

### 7.3 GPU 模型加载失败

原因：

- 模型目录不存在
- 显存不足
- `transformers` 版本不匹配

排查：

- 检查 `CHATTS_MODEL_PATH` / `QWEN_MODEL_PATH`
- 检查 GPU 显存
- 检查 `chatts` 环境依赖
