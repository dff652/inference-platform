## 二、与旧项目对接可行性

### 可直接复用

- API 模式：现有 FastAPI 路由结构可作为新平台蓝本
- 任务模型：Celery 任务链路与文档中执行适配器设计一致
- 数据模型：InferenceResult 等 SQLAlchemy 模型可迁移或映射
- 算法清单：ChatTS、Qwen-VL、ADTK-HBOS、Ensemble、Wavelet 等已明确

### 适配器层路径

```
新平台任务 → ExecutorAdapter → run.py CLI 调用
                              → 参数映射：--method/--input/--chatts_model_path/--task-id
                              → 结果解析：JSON annotations 格式
```

run.py 已支持 CLI 参数模式，适配器层通过 subprocess/Celery 调用完全可行，无需改动原项目。

## 三、主要风险

| 风险 | 级别 | 建议 |
|------|------|------|
| run.py 紧耦合（GPU环境/IoTDB/多算法混合） | 高 | 一期不动，通过 CLI adapter 绕过；二期逐步拆解 |
| 数据库共享冲突 | 中 | 一期独立 schema，通过视图/同步机制对齐旧表 |
| 模型环境隔离（ChatTS vs Qwen 不同 Python 环境） | 中 | 沿用现有多解释器方案，适配器层需感知 PYTHON_TRAINING_* 配置 |
| Celery broker 从 SQLite 迁移 | 低 | 生产环境已有 Redis compose 配置，直接切换 |
| 前端工作量 | 中 | P0 页面需 2-3 个完整页面开发 |

## 四、结论：完全可行

1. 架构风险低：一期"平台解耦 + 旧执行器适配"策略正确
2. 技术栈延续：FastAPI + Vue + Celery + SQLAlchemy，团队无学习成本
3. 接口契约明确：run.py 的 CLI 参数和 JSON 输出格式已标准化
4. 渐进式迁移：灰度方案保证了回退能力
5. 二期路径清晰：执行器抽象 → 资源池 → vLLM 集成
